# 司法拍卖信息自动抓取项目实施计划

> 本计划用于指导第一阶段开发：先在 Mac 当前项目中开发和测试，确认抓取结果正常后，再部署到局域网 Linux 电脑 `ecko@192.168.10.45` 定时运行。

## 一、总体目标

建立一个可以长期运行的自动化脚本，用于监控上海市浦东新区住宅司法拍卖信息。

第一阶段要做到：

- 每天自动抓取 2 次，时间为 11:00 和 23:00
- 抓取未来 45 天内开拍、以及正在进行中的住宅标的
- 保存每次抓取快照
- 维护每个标的从上拍到结束的生命周期数据
- 生成中文 Markdown 日报
- 输出 JSON 和 CSV 数据
- 支持手动运行和测试模式
- 后续可部署到 Linux cron 定时运行

## 二、是否需要操作界面

第一阶段 **不需要图形界面**。

原因：

- 当前规则数量不多，用配置文件更直接
- 图形界面会增加额外开发、维护和部署成本
- 第一阶段最重要的是抓取准确、数据保存完整、报告可读
- Linux 定时运行脚本时，配置文件比界面更稳定

第一阶段采用：

- `config/rules.json` 保存筛选规则
- `npm run fetch` 手动正常抓取
- `npm run fetch:test` 测试模式，只抓前 3 个标的
- Linux cron 每天 11:00 和 23:00 自动运行

后续如果脚本稳定运行 7 天以上，可以再考虑做一个本地配置界面。这个界面只用于修改筛选规则、查看报告，不做登录、报名、缴保证金、出价等任何拍卖操作。

## 三、技术路线

推荐使用：

- Node.js：脚本运行环境
- Playwright：打开网页并读取页面内容
- JSON：保存原始快照和生命周期数据
- CSV：方便 Excel/WPS 查看
- Markdown：生成每日阅读报告
- cron：Linux 上定时运行
- Vitest：测试价格、时间、地址、变化检测等核心逻辑

第一阶段先采用文件存储，不使用数据库。

原因：

- 数据量不大
- 文件容易检查和迁移
- 方便从 Mac 复制到 Linux
- 后续如果数据量变大，再考虑 SQLite

## 五、第一阶段开发任务

### 任务 1：初始化项目骨架

目标：让项目具备 Node.js 脚本运行和测试能力。

要做的事情：

- 创建 `package.json`
- 配置常用命令：
  - `npm run fetch`
  - `npm run fetch:test`
  - `npm test`
- 安装 Playwright
- 安装 Vitest
- 创建 `.gitignore`
- 创建 `README.md`

验收标准：

- 可以执行 `npm install`
- 可以执行 `npm test`
- 项目目录结构清晰

### 任务 2：创建规则配置文件

目标：把筛选条件放进配置文件，不写死在脚本里。

创建：

```text
config/rules.json
```

配置内容包括：

- 地区：上海市浦东新区
- 类型：住宅用房
- 时间范围：未来 45 天
- 每天抓取时间：11:00、23:00
- 价格档位：
  - 500 万以下
  - 500 万到 1000 万
  - 1000 万以上
- 排除关键词：
  - 商铺
  - 办公
  - 写字楼
  - 厂房
  - 工业
  - 土地
  - 单独车位
  - 车库
  - 仓储
  - 设备
  - 股权
  - 债权
  - 租赁权
- 围观人数明显变化规则：
  - 增加超过 100 人
  - 且涨幅超过 20%
- 测试模式数量：3 个标的
- 详情页访问间隔：1-3 秒
- 单次任务超时：60 分钟

验收标准：

- 脚本可以正确读取配置
- 配置缺少关键字段时能给出明确错误

### 任务 3：实现基础数据规范化

目标：把网页上不统一的文字转换成统一数据。

需要实现：

- 价格解析
  - 页面原文保留
  - 统一转换成“万元”
- 折扣率计算
  - 当前价 / 评估价
- 时间解析
  - `06月22日 10:00` 转成 `YYYY-MM-DD HH:mm:ss`
  - 注意跨年情况
- 地址处理
  - 保存标题原文
  - 尽量拆出小区、路名、门牌、楼栋室号
  - 生成地址指纹
- 状态标准化
  - 即将开始
  - 正在进行
  - 已成交
  - 流拍
  - 中止
  - 暂缓
  - 撤回
  - 变卖
- 风险关键词识别
  - 租赁
  - 占用
  - 腾退
  - 抵押
  - 查封
  - 瑕疵
  - 清场
  - 税费
  - 买受人承担

验收标准：

- 有单元测试覆盖价格、时间、地址、风险关键词
- 典型示例能解析正确

### 任务 4：实现网页抓取入口

目标：先实现一个稳定的数据来源入口，并预留另一个入口。

第一阶段数据源：

- 淘宝司法拍卖 / 阿里资产两个入口预留
- 实际先选择页面更稳定、字段更完整的一个入口

要做的事情：

- 用 Playwright 打开列表页
- 按规则筛选上海市浦东新区住宅用房
- 获取列表页标的
- 进入详情页读取基础字段
- 每个详情页之间随机等待 1-3 秒
- 详情页失败重试 1 次
- 列表页失败重试 1 次

详情页第一阶段抓取字段：

- 标题 / 地址
- 当前价 / 起拍价
- 评估价 / 市场价
- 开拍时间
- 结束时间 / 成交时间
- 状态
- 报名人数
- 围观人数
- 法院 / 处置单位
- 成交状态
- 成交价
- 竞买人数或报名人数
- 原始链接
- 标准化链接

不抓取：

- 图片
- PDF
- 附件
- 公告全文
- 竞买须知全文
- 标的介绍全文

验收标准：

- 测试模式能抓取前 3 个标的
- 正常模式能抓取当前符合条件的标的
- 不登录账号
- 不触发报名、出价等动作

### 任务 5：实现数据保存

目标：每次抓取都能留下可追溯的数据。

需要生成：

```text
data/latest.json
data/latest.csv
data/snapshots/YYYY-MM-DD_HH-mm.json
data/daily/YYYY-MM-DD.csv
data/lifecycle/auctions.json
data/lifecycle/assets.json
```

保存规则：

- 每次抓取保存一份快照
- `latest.json` 和 `latest.csv` 保存最新有效结果
- 每天保存一份 CSV 归档
- 生命周期数据按拍卖场次 ID 保存
- 疑似同一资产按地址指纹记录，但不自动合并

异常保护：

- 如果某次抓取数量明显异常，比如突然为 0，不覆盖 `latest`
- 异常抓取结果可以保存为快照或日志，方便排查

验收标准：

- 正常抓取后能生成 JSON、CSV
- 异常抓取不会污染最新数据
- 文件格式可以直接打开检查

### 任务 6：实现生命周期更新

目标：保留每个标的从首次发现到结束的全过程。

每个标的要记录：

- 首次发现时间
- 每次抓取快照
- 最新状态
- 首次上拍价格
- 当前价格
- 评估价
- 报名人数变化
- 围观人数变化
- 开拍时间变化
- 最终成交 / 流拍 / 撤回 / 中止 / 暂缓结果
- 成交价
- 成交时间 / 结束时间

识别规则：

- 同一场拍卖：用拍卖场次 ID
- 疑似同一套房：用地址指纹
- 地址相似只提示“疑似同一资产”，不自动合并

验收标准：

- 同一标的多次抓取会更新到同一生命周期记录
- 状态结束后仍然保留历史
- 重新上架的疑似同一资产可以被标记出来

### 任务 7：实现变化检测

目标：找出日报中需要提醒的变化。

第一阶段检测：

- 新增标的
- 撤回 / 中止 / 暂缓
- 状态变为正在进行
- 状态变为已成交 / 流拍 / 结束
- 起拍价 / 当前价变化
- 开拍时间变化
- 报名人数变化
- 围观人数明显变化

规则：

- 报名人数任何变化都提示
- 围观人数同时满足以下条件才提示：
  - 增加超过 100 人
  - 涨幅超过 20%
- 同一标的一天内多次报名变化，在日报里合并展示，例如 `0 -> 1 -> 2`

验收标准：

- 单元测试覆盖报名变化
- 单元测试覆盖围观人数明显变化
- 单元测试覆盖价格和状态变化

### 任务 8：生成 Markdown 日报

目标：生成一份中文、可直接阅读的每日监控报告。

日报文件：

```text
outputs/浦东住宅法拍每日监控_YYYY-MM-DD.md
```

日报结构：

1. 总览摘要
2. 最近 3 天新增标的
3. 当天重要变化
4. 未来 3 天即将开拍
5. 最近 7 天结束标的
6. 500 万以下
7. 500 万到 1000 万
8. 1000 万以上

总览摘要包括：

- 本次抓取时间
- 今日抓取次数
- 当前监控标的总数
- 新增标的数量
- 重要变化数量
- 报名人数变化数量
- 围观人数明显变化数量
- 最近 7 天结束数量
- 三个价格档位数量
- 抓取是否成功 / 是否有错误日志

每条房产显示：

- 标题 / 地址
- 当前价 / 起拍价
- 评估价
- 折扣率
- 开拍时间
- 状态
- 报名人数
- 围观人数
- 法院 / 处置单位
- 链接
- 风险备注，如有

验收标准：

- 日报是中文
- 不展示过多技术字段
- 三个价格档位正确
- 每个档位内按开拍时间从近到远排序
- 链接可点击

### 任务 9：实现主运行脚本

目标：把配置、抓取、保存、变化检测、报告生成串起来。

入口文件：

```text
scripts/run-fetch.js
```

运行命令：

```bash
npm run fetch
npm run fetch:test
```

主流程：

1. 读取配置
2. 创建必要目录
3. 启动日志
4. 打开浏览器
5. 抓取列表页
6. 筛选住宅标的
7. 抓取详情页
8. 标准化字段
9. 判断抓取结果是否异常
10. 保存快照
11. 更新生命周期数据
12. 检测变化事件
13. 写入 latest JSON / CSV
14. 更新当日 Markdown 日报
15. 写入运行日志

测试模式：

- 默认只抓前 3 个标的
- 不覆盖正式 `latest`
- 不更新正式日报
- 只用于开发和排错

验收标准：

- `npm run fetch:test` 可以安全运行
- `npm run fetch` 可以生成正式数据和日报
- 失败时有日志

## 六、开发顺序建议

推荐顺序：

1. 初始化项目骨架
2. 创建配置文件
3. 写价格、时间、地址、风险关键词解析
4. 写数据保存
5. 写生命周期和变化检测
6. 写日报生成
7. 接入 Playwright 抓取网页
8. 本地测试模式验证
9. 本地正式抓取验证
10. 部署到 Linux
11. 连续 7 天验收

这样做的好处是：先把数据结构、报告和变化检测做稳，再接真实网页。即使网页结构后面变化，也只需要集中调整抓取入口，不会影响后面的数据处理和报告逻辑。
