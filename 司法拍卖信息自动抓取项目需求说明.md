# 司法拍卖信息自动抓取项目需求说明

## 一、项目目标

建立一个长期运行的浦东新区住宅司法拍卖信息监控工具。系统每天自动抓取公开拍卖信息，生成中文日报，并保存完整历史数据，用于观察标的从上拍、状态变化、报名变化，到成交、流拍、撤回等全过程。

本项目只做公开信息整理和监控，不构成投资建议，不替代法律、税务、房产尽调或竞拍决策。

## 二、第一阶段监控范围

- 地区：上海市浦东新区
- 标的类型：住宅用房
- 保留范围：主体是住宅的标的，即使附带车位、地下人防车位，也保留
- 排除范围：商铺、办公、写字楼、厂房、工业、土地、单独车位、车库、仓储、设备、股权、债权、租赁权
- 处置阶段：即将开始、正在进行、已成交、流拍、中止、暂缓、撤回、变卖都纳入
- 数据来源：淘宝司法拍卖 / 阿里资产两个入口都预留，第一阶段只实现一个更稳定的入口
- 登录边界：不登录淘宝账号，只抓公开可见信息

脚本不得执行以下动作：

- 报名
- 缴纳保证金
- 出价
- 收藏或关注
- 绕过验证码或滑动验证
- 修改任何拍卖相关状态

## 三、抓取周期

- 抓取时间范围：按开拍时间判断，抓取未来 45 天内的标的
- 正在进行中的标的：即使开拍时间已过，也继续显示，直到成交、流拍或结束
- 抓取频率：每天 2 次
- 抓取时间：每天 11:00 和 23:00
- 详情页访问间隔：每个详情页之间随机等待 1-3 秒
- 单次任务超时：60 分钟
- 列表页访问失败：重试 1 次，仍失败则记录日志
- 详情页访问失败：重试 1 次，仍失败则记录该标的详情页失败

如果某次抓取结果明显异常，例如突然为 0 或远低于正常数量，系统应标记本次抓取异常，但不覆盖 `latest` 数据。

## 四、保存数据

输出三类文件：

- Markdown 日报，给人阅读
- JSON 数据，保存原始快照和生命周期
- CSV 数据，方便 Excel/WPS 打开分析

JSON 保存规则：

- 每次抓取保存一份原始快照
- 按拍卖场次 ID 维护生命周期数据
- 按地址指纹记录疑似同一资产关系，但不自动合并

CSV 保存规则：

- `data/latest.csv` 保存最新状态
- `data/daily/YYYY-MM-DD.csv` 保存每日归档

数据保留规则：

- 每次抓取快照永久保留
- 生命周期数据永久保留
- 每日 CSV 永久保留
- Markdown 日报永久保留
- 日志第一阶段永久保留，后续如文件过多再增加清理规则

第一阶段不做自动备份。

第一阶段不下载图片、PDF、附件，不保存公告、竞买须知、标的介绍等长文本。正常只保存结构化字段，解析异常时保存少量原始文本片段用于排查。

## 五、核心字段

每个标的尽量保存以下字段：

- 标题 / 地址
- 网站标题原文
- 详情页地址原文
- 小区、路名、门牌、楼栋室号等结构化地址
- 当前价 / 起拍价
- 页面原文价格
- 评估价 / 市场价
- 折扣率
- 开拍时间
- 结束时间 / 成交时间
- 状态
- 报名人数
- 围观人数
- 法院 / 处置单位
- 是否成交
- 成交价
- 竞买人数或报名人数
- 详情页原始链接
- 标准化详情页链接
- 来源入口
- 拍卖场次 ID
- 地址指纹
- 疑似同一资产标记
- 抓取时间
- 脚本版本
- 运行环境
- 单次抓取总耗时

价格统一保存为“万元”，同时保留页面原文。时间统一保存为 `YYYY-MM-DD HH:mm:ss`，页面只有月日时自动补全年份，并注意跨年判断。

## 六、日报规则

日报语言：中文，偏业务阅读，少技术字段。

日报周期：每天一份。上午抓取后生成或更新当天日报，晚上抓取后再次更新同一份日报。

日报开头包含总览摘要：

- 本次抓取时间
- 今日抓取次数
- 当前监控标的总数
- 新增标的数量
- 重要变化数量
- 报名人数变化数量
- 围观人数明显变化数量
- 最近 7 天结束数量
- 三个价格档位数量
- 抓取是否成功 / 是否有错误日志

日报按价格分三个档位：

- 500 万以下
- 500 万到 1000 万
- 1000 万以上

每个档位内按开拍时间从近到远排序。

每条房产显示：

- 标题 / 地址
- 当前价 / 起拍价
- 评估价
- 折扣率
- 开拍时间
- 状态
- 报名人数
- 围观人数
- 法院 / 处置单位
- 链接
- 风险备注，如有

日报额外区域：

- 最近 3 天新增标的
- 当天重要变化
- 未来 3 天即将开拍
- 最近 7 天结束标的

第一阶段不单独做高热度榜单。

## 七、变化事件

日报重点记录以下事件：

- 新增标的
- 标的撤回 / 中止 / 暂缓
- 状态变为正在进行
- 状态变为已成交 / 流拍 / 结束
- 起拍价 / 当前价变化
- 开拍时间变化
- 报名人数变化
- 围观人数明显变化

报名人数：任何变化都提示。

围观人数明显变化：相比上一次抓取，同时满足增加超过 100 人，且涨幅超过 20%。

同一标的一天内多次报名变化，日报中合并展示，例如 `0 -> 1 -> 2`。

## 八、生命周期逻辑

- 同一场拍卖用拍卖场次 ID 识别
- 同一套房可能流拍后重新上架，用地址指纹做疑似识别
- 地址相似时只标记“疑似同一资产”，不自动合并
- 标的一旦进入监控，即使后续暂缓、中止、撤回，或开拍时间移出未来 45 天，也继续保留生命周期数据
- 已不属于当前主列表的历史标的，不长期占据日报主列表，只在重要变化或最近结束区域出现

## 九、风险备注

第一阶段做简单关键词风险备注，不做法律判断。

关键词包括：

- 租赁
- 占用
- 腾退
- 抵押
- 查封
- 瑕疵
- 清场
- 税费
- 买受人承担

风险备注不影响排序，只作为提醒显示。

## 十、运行方式

开发流程：

- 先在 Mac 当前项目中开发和测试
- 确认抓取结果正常后，部署到局域网 Linux 电脑 `ecko@192.168.10.45`

定时方式：

- 第一阶段使用 cron
- 每天 11:00 和 23:00 自动运行
- 后续如需要更稳定的任务状态管理，再升级为 systemd timer

运行模式：

- 支持手动运行一次
- 手动运行也默认按配置文件执行
- 支持测试模式
- 测试模式默认只抓前 3 个标的，不覆盖正式数据，不更新正式日报

## 十一、配置方式

筛选条件放入配置文件，例如：

```text
config/rules.json
```

配置内容包括：

- 地区
- 标的类型
- 未来 45 天
- 排除关键词
- 价格档位
- 抓取时间
- 围观人数变化阈值
- 新增标的展示天数
- 最近结束展示天数
- 测试模式数量

## 十二、第一阶段成功标准

第一阶段完成后，应满足：

- 能连续稳定抓取 7 天
- 无明显漏抓或重复异常
- 能正确生成 Markdown 日报、JSON、CSV
- 能看出新增、状态变化、报名变化、围观人数明显变化
- 能完整跟踪至少一批标的从上拍到结束的过程
- Mac 上可手动运行测试
- Linux 上可用 cron 定时运行
